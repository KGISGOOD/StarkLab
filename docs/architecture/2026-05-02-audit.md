## 2. 主要架構問題（依風險排序）

### 🔴 HIGH — 立即止血

| # | 問題 | 風險說明 |
|---|------|---------|
| 1 | `ngrok.exe`、`w.db`、`*.csv` 在 git repo 中 | 資料庫和敏感資料洩漏 |
| 2 | `@csrf_exempt` 使用於 3 個公開 endpoint 無任何保護 | CSRF 攻擊 + 濫用風險 |
| 3 | 缺少 `SECURE_SSL_REDIRECT` / `SESSION_COOKIE_SECURE` / HSTS | 中間人攻擊風險 |
| 4 | CSV 使用相對路徑（`'chat_records.csv'`） | 路徑相依，部署環境不同即失效 |
| 5 | `LANGUAGE_CODE` / `TIME_ZONE` 未設定為正確值 | 時間戳記混亂、i18n 問題 |

### 🟠 MEDIUM — 技術負債收斂

| # | 問題 | 影響 |
|---|------|------|
| 6 | `urls.py` 含大量死碼註解 | 可讀性差，維護困難 |
| 7 | `project_wra_repoet` typo URL | 使用者存取到壞掉的 URL |
| 8 | 三份 static 目錄同時存在（`mylab/static/`、`lab/static/`、`staticfiles/`）| collectstatic 行為混亂 |
| 9 | `project2_views.py` 1560 行、`project3_views.py` 1320 行 | 單檔過大，難以 review 和測試 |
| 10 | `Templates/` 目錄不符 Django 規範（應為 `templates/mylab/`）| 部分環境找不到模板 |
| 11 | SQLite 在 Windows 多程序環境下有並發鎖問題 | 高流量時資料寫入失敗 |

### 🟡 LOW — CSS 設計負債

| # | 問題 | 影響 |
|---|------|------|
| 12 | 14 個 CSS 檔案無設計 token，顏色/字級散落各處 | 維護成本高，品牌一致性差 |
| 13 | 同 selector 在多檔重複定義、存在 !important 衝突 | 難以預測樣式覆蓋行為 |
| 14 | 無元件化結構 | 無法複用，每頁樣式獨立修改 |

## 3. 建議修正步驟（Sprint 1 規劃基礎）

### Phase 1 — 止血（HIGH，必須先做）
1. 更新 `.gitignore`，移除敏感檔案
2. 修正 `settings.py`（語系/時區/安全標頭）
3. @csrf_exempt endpoint 加 token + rate limit
4. CSV 路徑改用 `BASE_DIR`

### Phase 2 — 收斂（技術負債）
5. 清理 `urls.py`，加 typo redirect
6. Static 目錄收斂
7. 大型 views 拆檔（不改邏輯）
8. Templates 目錄規範化
9. 評估 SQLite → PostgreSQL

### Phase 3 — CSS 設計基線
10. T1：全面盤點現有 CSS（14 個檔案）
11. T2：設計三層架構重構計畫書
12. （下個 Sprint）實際執行 CSS 重構
